# stampme — Plan 1: Foundation

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bootstrap stampme project — git, Next.js + TS strict, Tailwind/shadcn RTL, Prisma + Neon Postgres, Clerk auth, i18n (ar/en), observability stack, Arabic landing page, deployed to Vercel.

**Architecture:** Single Next.js 15 monolith with App Router. Multi-tenant Postgres via Prisma. Clerk handles auth + SMS OTP. Marketing landing at `/`, dashboard at `/(merchant)/*`, public enrollment at `/c/[slug]`, scanner at `/scan` (later plans).

**Tech Stack:** Next.js 15, TypeScript strict, Tailwind + shadcn/ui, Prisma, Postgres (Neon), Clerk, next-intl, Zod, Sentry, PostHog, Resend, Cloudflare R2, Vercel, bun

**Spec reference:** `docs/superpowers/specs/2026-04-28-stampme-design.md`

  ```ts
  import { NextResponse, type NextRequest } from "next/server";
  import { Webhook } from "svix";
  import slugify from "slugify";
  import { db } from "@/lib/db";
  import { env } from "@/lib/env";
  import { clerkUserCreatedSchema } from "@/lib/validation";

  export const runtime = "nodejs";

  // Generate a unique merchant slug from the inferred name. Slug appears in
  // public URLs `/c/[merchantSlug]` (Plan 4) so it must be URL-safe and unique.
  // Plan 2 hardens this with collision-suffix logic in `src/lib/slug.ts`; here
  // we use a minimal inline version since the wizard runs after webhook fires
  // and will overwrite name/slug if needed.
  function slugForMerchant(name: string, fallback: string): string {
    const base = slugify(name, { lower: true, strict: true, locale: "ar" }) || fallback;
    return base;
  }

  export async function POST(req: NextRequest) {
    const svixId = req.headers.get("svix-id");
    const svixTimestamp = req.headers.get("svix-timestamp");
    const svixSignature = req.headers.get("svix-signature");

    if (!svixId || !svixTimestamp || !svixSignature) {
      return new NextResponse("Missing svix headers", { status: 400 });
    }

    const rawBody = await req.text();
    const wh = new Webhook(env.CLERK_WEBHOOK_SIGNING_SECRET);

    let event: unknown;
    try {
      event = wh.verify(rawBody, {
        "svix-id": svixId,
        "svix-timestamp": svixTimestamp,
        "svix-signature": svixSignature,
      });
    } catch (err) {
      console.error("[clerk webhook] signature verification failed", err);
      return new NextResponse("Invalid signature", { status: 401 });
    }

    // We only care about user.created here; expand later.
    const parsed = clerkUserCreatedSchema.safeParse(event);
    if (!parsed.success) {
      // Other event types arrive at this endpoint too — ack 200 to stop retries.
      return NextResponse.json({ ignored: true });
    }

    const { id: clerkUserId, email_addresses, primary_email_address_id, phone_numbers, primary_phone_number_id, first_name, last_name } =
      parsed.data.data;

    const primaryEmail =
      email_addresses.find((e) => e.id === primary_email_address_id)?.email_address ??
      email_addresses[0]?.email_address;
    const primaryPhone =
      phone_numbers.find((p) => p.id === primary_phone_number_id)?.phone_number ??
      phone_numbers[0]?.phone_number ??
      "";

    if (!primaryEmail) {
      return new NextResponse("No primary email", { status: 400 });
    }

    const inferredName =
      [first_name, last_name].filter(Boolean).join(" ").trim() || primaryEmail.split("@")[0]!;

    // Idempotent upsert by clerkUserId. Slug must be unique on insert; Plan 2
    // re-runs slug generation with collision suffix when the wizard saves.
    const baseSlug = slugForMerchant(inferredName, clerkUserId.slice(-8).toLowerCase());
    let slug = baseSlug;
    for (let i = 0; i < 5; i++) {
      const existing = await db.merchant.findUnique({ where: { slug } });
      if (!existing) break;
      slug = `${baseSlug}-${Math.random().toString(36).slice(2, 6)}`;
    }

    await db.merchant.upsert({
      where: { clerkUserId },
      update: {},
      create: {
        clerkUserId,
        name: inferredName,
        slug,
        ownerEmail: primaryEmail,
        ownerPhone: primaryPhone,
        vertical: "OTHER",
        brandColor: "#0A7C36",
      },
    });

    return NextResponse.json({ ok: true });
  }
  ```

  Why upsert + `OTHER` vertical: signup happens before the onboarding wizard (Plan 2). The wizard updates `name`, `vertical`, `brandColor`, `logoUrl`. We just need a row to exist so the dashboard middleware can resolve `merchantId` from `clerkUserId`.

## Performance & bundle notes (for future plans to respect)

- **Date libs:** Do NOT install `moment` (heavy) or `date-fns` (≈ 75 KB) for the landing. We're using native `Intl.DateTimeFormat` and `new Date()` only. Plans 2+ that need date math should justify with bundle impact.
- **Icon libs:** `lucide-react` is tree-shakable per icon — only import what you need (`import { Check } from "lucide-react"`). Never `import * as Icons`.
- **PostHog:** the `posthog-js` browser bundle is ~30 KB gzipped. Do not include it on the marketing landing — it's loaded only inside the merchant dashboard (Plan 7) and the customer enrollment page (Plan 4).
- **Server components by default:** the only `"use client"` files in this plan are `LocaleSwitcher.tsx` and Sentry's `instrumentation-client.ts`. Keep this discipline going forward.

## Security notes (for future plans)

- All `Merchant` row queries from the merchant dashboard MUST filter by `clerkUserId === auth().userId` derived in the request, not from the request body. Never trust client-supplied `merchantId`.
- The Clerk webhook endpoint is the ONLY place a `Merchant` row is created with arbitrary `clerkUserId`. Other places must look it up from the authenticated session.
- Postgres RLS (mentioned in spec §6) is deferred to a later plan — first MVP relies on app-layer enforcement. Document this trade-off when implementing dashboards (Plan 7).
- CSP headers + Vercel security headers are out of scope for Plan 1; add in a future hardening plan.
